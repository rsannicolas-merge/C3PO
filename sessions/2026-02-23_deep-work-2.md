## What Worked

Preserve these approaches.

- **Parallel reformat agents (6 simultaneous)**: All 6 completed successfully with detailed structural mapping reports. No manual cleanup needed. Each agent produced a structural change summary showing exactly what moved where.
- **PDF template review before reformatting**: Reading the 3 official templates first gave agents a precise target. Without this, reformats would have been speculative.
- **Background agents + foreground work**: While 6 reformat agents ran in background, completed 3 other tasks (policy-draft command, systems.md ingestion, policy library update) in foreground. Maximum throughput.
- **Operational intelligence extraction from playbooks**: The Google SecOps Playbook PDF was a goldmine -- it contained confirmed tool inventory, MSSP contacts, infrastructure details, and severity SLAs that were all listed as "unknown" or "to be inventoried" in systems.md. One document closed dozens of inventory gaps.

## Key Learnings

*Generalizable insights -- candidate entries for memory/learnings.md*

1. **Source official templates before generating drafts**: Reading the canonical template first saves a full reformat pass later. In this session, 6 drafts needed structural reformat because they were generated against an informal template. If the MERGE PDFs had been reviewed first, the initial `/policy-draft` output would have matched.
   - Applies when: Any batch document generation where an official template exists
   - Evidence: 6 agents x ~5 min each = ~30 min of reformat work that could have been avoided
   - Confidence: 9/10

2. **Operational playbooks are the richest source of asset inventory data**: The Google SecOps Playbook contained more confirmed, actionable system data than any other single source reviewed to date -- tool names, versions, IP addresses, contacts, SLAs, and domain lists. Playbooks are written by operators who know what's deployed.
   - Applies when: Building or updating a security asset registry from scratch
   - Evidence: systems.md went from 9 tools to 14, 1 vendor to 7, zero infrastructure entries to 3, zero SLAs to 4 severity tiers -- all from one playbook PDF
   - Confidence: 9/10

3. **Definitions sections pay for themselves in audit**: Adding 10-15 defined terms per document took minimal effort but provides significant auditor clarity. HITRUST assessors evaluate whether the organization understands its own controls -- defined terms demonstrate that.
   - Applies when: Any compliance document destined for external review
   - Evidence: All 6 reformatted documents now have Definitions sections with 10-15 terms each
   - Confidence: 8/10

## Domain Insights

*Anything specific to this system, vendor, framework, or process*

- MERGE TechOps uses a consistent 9-section template across all document types (Policy, Procedure, Playbook). Section 5 is the only one that changes name: "Policy Overview" / "Standard Requirements" / "Procedure Overview" / "Playbook Overview"
- Procedure template adds a Troubleshooting section (Section 6) not present in Policy template -- this is where escalation paths and common issue resolution live
- Playbook template includes RACI matrix in Roles section and operational subsections (Critical URLs, Emergency Contacts, Daily Ops Checklist, Severity Classification) -- these are the operational quick-reference elements
- Foresite MSSP operates on a 4-tier severity model with 15-min Critical response SLA -- this is the baseline for all IR policy drafts
- Google SecOps uses Workforce Identity Federation (not service accounts) for SSO -- secops-entra-pool is the workforce pool name
- 26 MERGE domains are monitored via Google Threat Intelligence -- this is the external attack surface being watched
