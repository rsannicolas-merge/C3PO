## What Worked

Preserve these approaches.

- **Parallel policy generation**: 5 agents producing drafts simultaneously. Total wall-clock time ~10 minutes vs. ~50 minutes sequential. Each agent had clear scope and the same format reference (POL-0011).
- **Format-reference-first approach**: Giving each agent the existing approved policy as a template ensured structural consistency across all 5 outputs without manual reformatting.
- **Plan-then-execute workflow**: Using plan mode to design the 7-file integration before touching any files prevented rework and ensured all cross-references were consistent.
- **Policy hierarchy design**: POL → STD → PROC hierarchy (policy sets intent, standard sets requirements, procedure sets steps) maps cleanly to HITRUST evidence structure.

Document for future reference.

- **Background agents from previous context**: Agents launched before context compaction didn't write their files. Had to detect the gap (directory listing showed only POL-0011) and re-launch a fresh batch. Lesson: verify agent outputs explicitly after context recovery.

## Key Learnings

*Generalizable insights -- candidate entries for memory/learnings.md*

1. **Parallel agent policy generation scales well**: 5 independent policy drafts generated simultaneously in ~10 minutes. Key enabler: each draft has the same format template and no cross-dependencies. This pattern applies to any batch of independent documents that share a common structure.
   - Applies when: Generating multiple independent policy/procedure/standard documents
   - Evidence: 5 agents, 5 drafts, ~2,300 lines total, all structurally consistent
   - Confidence: 9/10

2. **DevSecOps policies need AI-specific sections for HITRUST AI2**: Each DevSecOps document (pipeline, container, IaC, dependency, SDLC) needs explicit AI/ML subsections to satisfy HITRUST AI2 requirements. AI lifecycle controls don't live solely in the AI Governance Policy -- they permeate the SDLC.
   - Applies when: Drafting any security policy/standard for an organization with AI systems in scope for HITRUST
   - Evidence: HITRUST AI2 control 09.ab.05 (AI Model Security) maps to pipeline, container, and IaC standards, not just AI governance
   - Confidence: 8/10

3. **Policy hierarchy (POL → STD → PROC) maps to HITRUST evidence tiers**: HITRUST requires policy (intent), implementation (technical standard), and procedure (operational steps) evidence per control specification. Structuring documents as POL/STD/PROC creates natural 1:1 mapping to evidence requirements.
   - Applies when: Designing policy library structure for any HITRUST assessment
   - Evidence: POL-0012 provides policy coverage for SA-3; STD-0001/0002/0003 provide implementation evidence; PROC-0001 provides operational procedure evidence
   - Confidence: 8/10

## Domain Insights

*Anything specific to this system, vendor, framework, or process*

- NIST SA-3 (System Development Life Cycle) is the anchor control for DevSecOps -- everything else (SA-11, SA-15, CM-3, SI-7) branches from it
- HITRUST CSF v11.5 control 09.aa (Change Control Procedures) is the primary DevSecOps evidence target -- it requires documented pipeline security, code review, and deployment gates
- Container security standards must address both build-time (Dockerfile, base images, scanning) and runtime (network policies, resource limits, security contexts) -- HITRUST expects both
- IaC security is a force multiplier for HITRUST evidence: if security baselines are codified in Terraform, the Terraform configs themselves become compliance evidence
